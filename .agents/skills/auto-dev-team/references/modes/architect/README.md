# Architect 模式 (新功能开发)

> 适用: 用户想开发新功能、实现新需求 | 产出: `current-steps.md`；命中大测试时再产出 `current-test.md`

## 目录

- AI 必须主动读取
- 流程
- 小改动变体
- 强制检查
- 模式切换
- 菜单输出约定
- 阶段结束选项
- Phase 4: 进入 Step 执行

## AI 必须主动读取

进入此模式时，AI 必须主动读取以下文件（无需用户提供）：

- `.autodev/project-map.md` - 了解项目结构
- `.autodev/module-registry.md` - 查找可复用组件

## 流程

### Phase 0: 苏格拉底问诊（需求探索）

> ⚕️ 先问清楚再动手——不问诊就开方是庸医

**触发条件**：需求描述模糊（< 3 句话，或含“做一个”“搞个”“优化一下”等模糊词）

**跳过条件**：需求已含具体场景 + 边界条件 + 验收标准，或用户说“直接做”

```text
AI:   0. 主动读取 project-map.md 和 module-registry.md
      1. 苏格拉底式提问（每次只问一个）
         - 目的: 你要解决什么问题？
         - 用户: 谁会用？什么场景下用？
         - 边界: 有什么限制？（时间 / 技术栈 / 兼容性）
         - 成功: 做成什么样算完成？
      2. 优先用选择题，降低回答成本
      3. 需求足够清晰后 → 进入 Phase 1
```

⛔ 每条消息最多问 1 个问题
⛔ 先问目的，再问方案

### Phase 1: 需求确认 + 行为场景包

```text
用户: [经 Phase 0 澄清后的需求] / [本身已清晰的需求]

AI:   0. （若 Phase 0 已读取则跳过）主动读取 project-map.md 和 module-registry.md
      1. 重述需求（专业简洁）
      2. 生成行为场景包：
         - 用户明确提出的场景
         - AI 自动补充的通用负例 / 错误链路 / 边界 case
         - 待业务确认的规则型场景（如阈值、权限、重试上限）
      3. 若项目已有 BDD 框架，且场景适合用 `.feature` 沉淀：
         - 路径: features/{功能名}.feature
         - 无法自动化的场景标注 @manual
      4. 标注是否命中 GUI-capable task（页面流程 / 跳转 / 会话 / 权限 / 表单 / 任意可交互界面）
      5. "场景覆盖全了吗？需要补充吗？"

用户: 确认 / 补充场景 / 回答待确认问题
```

### Phase 2: 可行性评估

```text
AI:   0. 测试环境检查:
         - 后台自动测试框架（单测 / 集成 / 契约）
         - BDD 框架（如有）
         - GUI executor（Playwright / 桌面 GUI driver / 宿主操作入口 等，如有）

      1. 查 module-registry: "搜索 [关键词]: 找到 xxx / 未找到"

      1.5 🏠 归属判断（强制，在设计新结构前）:
          - 这个功能在语义上属于哪个领域？（用户、订单、通用工具...）
          - 项目中是否已有处理该领域的模块？
            ├── ✅ 有 → 评估：融入现有模块（扩展）还是独立新建？为什么？
            └── ❌ 无 → 确认需要新建模块

          📐 模块策略（可组合，一个功能可能涉及多种策略）:
          ├── [融入] 扩展 {现有文件}（新增什么）
          ├── [复用] 调用 {现有组件/工具}（列出复用项）
          └── [新建] 创建 {新文件}（理由、职责）

      2. 影响范围分析: 🟢小 / 🟡中 / 🔴大
         - 优先执行 `scripts/blast-radius.py` 生成候选影响面
         - 调用方 / 消费方 / 契约兼容性
         - 地址 / 路径 / 配置 / 数据格式变化

      3. 边界 case:
         - 异常输入？并发？空数据？大数据？重试？权限？

      4. 测试等级判断:
         - 小测试：单模块、低风险、非关键链路
         - 大测试：认证 / 支付 / 权限 / 多步流程 / 跨模块 / 契约变更 / 业务规则不清
         - 是否需要 `.autodev/current-test.md`
         - 是否需要进入 GUI 自治验收闭环
         - GUI executor、可视化执行策略、失败后自修复轮次
         - 观测驱动验证档位：行为改动默认至少 `L1`；跨模块 / 状态流转 / 异步 / 权限等场景升到 `L2`

      5. 🔷 抽象潜力评估（三问法则）
      6. 判断: 💗更优方案 / ⚠️有风险 / ✅可行
```

### Phase 3: 生成计划（增量可测拆分）

```text
AI:   1. 评估复杂度:
         - [极简] 1步 → 建议一键完成
         - [简单] 2-3步
         - [中等] 4-5步
         - [复杂] >5步 → 建议拆分需求

      2. 拆分原则（关键）:
         - 每步必须产出可独立验证的模块
         - 每步都要写明覆盖哪些场景
         - 每步都要写明后台自动测试如何跑
         - 每步都要写明观测驱动验证档位与主观测面
         - 命中 GUI-capable task 的步骤，必须写明 GUI executor、可视化执行策略、GUI case matrix
         - 若当前步骤尚不具备 GUI 联调条件，必须明确写“暂不可执行”

      3. 计划必须落实 Phase 2 的模块策略:
         - 每步标注目标文件（融入还是新建）
         - 每步标注 Blast Radius 目标（文件 / 符号）与预期风险阈值
         - 每步标注覆盖场景
         - 每步标注后台自动测试方式
         - 每步标注观测驱动验证档位、主观测面、备用观测面
         - GUI 步骤标注 executor、可视化执行策略、关键 GUI case、Gate 结论

      4. 写入 `current-steps.md`，包含:
         - Log 标识: [DEV-{任务主题}]
         - 关键决策
         - 测试等级: [小测试 / 大测试]
         - 测试台账路径: [无 / .autodev/current-test.md]
         - 每步的目标文件 + 覆盖场景 + 后台自动测试
         - 每步的 Blast Radius 目标 + 风险阈值 + 报告路径
         - 每步的观测驱动验证档位 + 主/备用观测面
         - 每步的 GUI 自治验收信息（是否命中 / executor / 可视化执行 / 关键 case / Gate）
         - 每步的 AI 执行版 How to Test（预期观测 + 通过标准）

      5. 若命中大测试:
         - 初始化 `.autodev/current-test.md`
         - 写入场景矩阵、待业务确认问题、初始测试计划

      6. "Planning ready. Waiting for ❇️"
```

提醒：

- 执行时每步使用 `[DEV-{主题}-Step{N}]` log 追踪流程
- 执行时每步在第一行代码前都要先跑一次 Blast Radius
- 每步必须能独立验证，禁止积攒到最后
- 新功能不是“等失败再观测”，每个有行为改动的 Step 默认至少执行一轮 `L1` 观测驱动验证
- GUI-capable task 在计划阶段就要显式标出来，避免后面才想起界面验收

### Phase 3.5: PM 摘要（自适应）

**触发条件**：用户使用业务语言描述需求时自动附加

```text
📋 计划摘要

做什么: [一句话业务语言描述]
分几步: N 步
预计改动: X 个文件
风险等级: 🟢低 / 🟡中 / 🔴高
测试等级: 小测试 / 大测试
GUI 自治验收: 不触发 / 默认执行 / 暂不可执行
可回退: ✅ 随时可以撤销
```

**跳过条件**：用户使用技术语言描述，或主动说“不用解释”

## 小改动变体

如果改动很小（≤2 文件，≤30 行）→ 跳过计划，直接评估后一键实现（自动切换到 FastTrack 模式）

## 强制检查

- 单文件职责过多或体积明显膨胀 → 建议拆分，提醒用户
- 发现可复用组件 → 必须复用，不得新建
- 发现现有模块可扩展 → 优先融入，不得重复新建
- 文档缺失 → 告知用户，建议先 Survey
- 业务规则不清 → 必须停下来问，不能先把场景写死为通过标准

## 模式切换

发现需要大量重构 → “建议先用 Refactor 模式处理 xxx，再回来做新功能”

## 菜单输出约定

- 菜单型 UI 遵循 `references/shared/menu-contract.md`
- 阶段推进优先复用 `references/shared/flow-snippets.md` 中的 `phase-confirm` 模板
- 会诊后的执行分叉优先复用 `references/shared/flow-snippets.md` 中的 `plan-choice` 模板

## 阶段结束选项

### Phase 0 结束

```text
📍 当前: 需求已探索清楚
📌 下一步:
[1] 确认 - 进入需求确认，生成行为场景包
[0] 取消
```

### Phase 1 结束

```text
📍 当前: 需求已理解为"[复述需求]"
📌 下一步:
[1] 确认 - 进入可行性评估，分析影响范围和测试等级
[2] 补充 - 如需补充: [列出缺失点，如边界情况 / 权限规则 / 失败策略]
[0] 取消
```

### Phase 3.9: ⭐ 自动会诊（强制）

方案输出后，按 `references/principles/critique.md` 的模式集成流程执行自动会诊。
本模式传递：【用户原始需求】+ 计划内容、影响范围、关键决策、测试等级。

### Phase 3 结束

```text
📍 当前: 计划已生成，共 N 步，预计改动 X 个文件

━━━━━━━━━━━━━━━━━━━━
🏥 会诊报告（Critique Subagent）
━━━━━━━━━━━━━━━━━━━━
[Subagent 自动审查结果]
━━━━━━━━━━━━━━━━━━━━

📌 下一步:
[1] 执行原计划（进入 auto-dev-team/step 流程）- 💿 自动快照，逐步执行，每步 1 个 💾 存档
[2] 执行修订计划（进入 auto-dev-team/step 流程）- 💿 自动快照，逐步执行，每步 1 个 💾 存档
[3] 信任模式（进入 auto-dev-team/step 流程）- 🎯 里程碑 + 💿 快照后连续执行，完成后 1 个 💾 存档
[0] 取消
```

## Phase 4: 进入 Step 执行

实际执行统一交给 `references/modes/step/README.md`。

进入 Step 前，必须确保：

- 共享写前置已按 `references/write-preflight.md` 执行。
- 若本计划会创建或修改 `.feature` / step definitions，先读 `references/principles/bdd-testing.md`。
- 若本任务为大测试，先读 `.autodev/current-test.md`。
- Step 阶段的影响分析、后台自动测试、GUI 自治验收闭环、存档、信任模式（含里程碑）与任务收尾，均以 `references/modes/step/README.md`、`references/principles/test-verification.md`、`references/principles/gui-autonomous-loop.md` 和 `references/principles/checkpoint-mechanism.md` 为准。

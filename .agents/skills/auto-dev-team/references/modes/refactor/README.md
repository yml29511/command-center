# Refactor 模式 (代码重构)

> 适用: 代码太乱、想整理、想重构 | 产出: current-steps.md

## 目录

- AI 必须主动读取
- 流程
- 强制规则
- 抽象提取重构 (特殊类型)
- 等价性验证
- 模式切换
- 菜单输出约定
- 阶段结束选项

## AI 必须主动读取

进入此模式时，AI 必须主动读取以下文件（无需用户提供）：
- `.autodev/project-map.md` - 了解项目结构
- 用户提到的目标文件

## 流程

### Phase 1: 异味识别
```
用户: [描述重构目标]

AI:   0. 主动读取 project-map.md 和目标文件

AI:   分析并输出:
      
      🔍 代码异味:
      - [ ] 文件过长 (>300行)
      - [ ] 函数过长 (>50行)
      - [ ] 职责混乱
      - [ ] 重复代码 → 考虑抽象提取
      - [ ] 命名不清
      
      🔷 抽象提取机会 (如适用):
      - [ ] 发现可抽象的中性逻辑
      - [ ] 相同模式出现 3+ 次
      
      如果是"抽象提取"类型重构:
      → 用三问法则评估:
         Q1: 中性动作 or 业务特化?
         Q2: 未来其他场景会用吗?
         Q3: 抽象后 API 更简单吗?
         参考: `references/principles/abstraction-rules.md`
```

### Phase 2: 影响范围分析
```
AI:   1. 优先执行 `scripts/blast-radius.py` → 输出报告路径、风险等级、直接调用点
      2. 扫描直接调用点 → 列出 文件|行号|场景|风险
      3. 检查间接影响 (第2层依赖)
      4. 影响范围评估: 🟢小 / 🟡中 / 🔴大
      
      "影响范围: [大/中/小]，确认继续？"
```

### Phase 3: 方案设计
```
AI:   📋 重构方案:
      1. [具体改动]
      2. [具体改动]
      
      🛡️ 向后兼容策略:
      - [如何保证旧调用不挂]
      
      ✅ 回归测试清单:
      1. [验证点1]
      2. [验证点2]
      
      "确认方案？"
```

### Phase 4: 渐进执行 (增量可测)
```
AI:   生成 Step 计划，写入 current-steps.md，包含:
      
      ## 关键决策 (防遗忘)
      - **向后兼容策略**: [具体策略]
      - **回归测试点**: [每步后要验证什么]
      - **Blast Radius 策略**: [每步目标 / 风险阈值 / 报告路径]
      - **观测驱动验证档位**: [默认 L3]
      - **观测标识**: [REFACTOR-{目标}]
      
      执行要求:
      - 每步只改一个模块
      - 每步必须保持系统可运行 (增量重构)
      - 每步先采 before 基线，再采 after 观测
      - 每步第一行代码前先刷新 Blast Radius
      - 每步后立即验证回归清单 (通过 before / after 观测对比)
        ⚠️ 禁止 "Step 1: 提取函数, Step 2: 测试" → 应该 "Step 1: 提取函数并验证等价"
      - 每步产出可测标准: 重构前后输入 / 输出 / 分支 / 副作用保持一致
      - 最后一步清理 [REFACTOR-*] 临时观测
      - 完成后自动更新 project-map.md
        输出: "📝 已自动更新: project-map.md - [模块变化简述]"
```

## 强制规则

- 必须输出调用清单
- 必须说明向后兼容策略
- 必须列出回归测试点
- 高风险调用点需用户二次确认
- 抽象提取后必须更新 module-registry.md

## 抽象提取重构 (特殊类型)

```
当 Refactor 目标是"抽象提取"时:

1. 识别所有重复点
2. 设计统一 API (遵循三问法则)
3. 提取到独立文件/模块
4. 替换所有调用点
5. 更新文档:
   - module-registry.md (新增 Utility)
   - project-map.md (标注新通用能力)

输出: "📝 已自动更新: module-registry.md - 新增 [Utility名]"
```

## 等价性验证

```
重构完成后，必须通过观测对比验证:
1. 原有功能行为完全不变
   → 对比重构前后输入 / 输出 / 分支 / 副作用是否一致
2. 原有调用方式仍可用
   → before / after 基线显示所有调用路径正常
3. 无新增副作用
   → 无额外写库、请求、状态变更等副作用
```

## 模式切换

如发现需要新功能支撑重构 → "建议暂停，先实现 xxx 功能后再继续重构"

## 菜单输出约定

- 菜单型 UI 遵循 `references/shared/menu-contract.md`
- 阶段推进优先复用 `references/shared/flow-snippets.md` 中的 `phase-confirm` 模板
- 会诊后的执行分叉优先复用 `references/shared/flow-snippets.md` 中的 `plan-choice` 模板

## 阶段结束选项

### Phase 2 结束
```
📍 当前: 影响范围分析完成，共影响 [N] 个文件，风险 [🟢/🟡/🔴]
📌 下一步:
[1] 确认 - 进入方案设计，制定重构策略
[2] 缩小范围 - 减少本次重构涉及的文件
[3] 详情 - 查看具体调用点和风险明细
[0] 取消
```

### Phase 3.9: ⭐ 自动会诊（强制）

方案输出后，按 `references/principles/critique.md` 的模式集成流程执行自动会诊。
本模式传递：【用户原始重构目标】+ 重构方案、影响范围、向后兼容策略。

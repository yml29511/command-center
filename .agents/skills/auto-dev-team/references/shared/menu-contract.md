# Menu Contract

> 菜单型 UI 的统一协议。目标是让交互更稳定、更易回复，同时保持上下文贴合，不做 railroading。

## 目标

- 让用户能快速扫读并做出下一步选择
- 让不同模式下的菜单保持一致的结构和语气
- 让模型优先输出高信号选项，而不是机械堆选项

## 总原则

- 菜单是交互协议，不是装饰文本
- 只有在需要用户做明确分叉、确认、取舍时，才输出菜单
- 若只需要用户补一个事实、回答一个问题，直接提问，不输出菜单
- 选项数量应与当前阶段和真实分叉贴合
- 优先提供少量高信号、彼此有区分度的选项；复杂场景可适当增加
- 不得为了“引导”而隐藏合理替代路径；若存在关键分叉，应如实给出
- 用户的自然语言意图优先于菜单数字

## 固定格式

- 第一行使用：`📍 当前: ...`
- 第二行使用：`📌 下一步:`
- 每个选项单独一行，格式为：`[N] 动作 - 结果 / 目的`
- 选项文案用动词开头，如：`确认`、`补充`、`执行`、`继续`、`切换`、`结束`
- 风险提示、背景说明放在菜单前，不放进选项正文里
- 不在菜单块中混入大段解释

## 编号规则

- `[0]` 保留给：`取消 / 结束 / 暂停`
- 其余编号按推荐顺序从 `[1]` 递增
- 最推荐、最符合当前流程的动作放在前面
- 若存在“继续当前流程”和“切换模式”的分叉，默认先列继续当前流程

## 文案规则

- 每个选项只表达一个动作
- 动作后可补一句短说明，但不要展开成长段落
- 避免两个选项语义重叠
- 若某个选项依赖条件，必须把条件说清楚

## 省略规则

- 没有修订计划时，不出现“执行修订计划”
- 没有明确的信任模式上下文时，不出现“信任模式”
- 没有具体可展开内容时，不出现“详情”“深入”“查看明细”
- 没有明确切换目标时，不出现“切换到更合适的模式”

## 用户回复解析

- 用户回复数字时，按菜单编号理解
- 用户回复自然语言时，按真实意图理解，优先于编号
- 用户回答未在菜单中列出的合理动作时，可直接按该动作继续，不强迫重选
- 若用户回复存在歧义，再追问一次最小必要澄清

## 菜单家族

- `phase-confirm`
  - 用于阶段确认、推进到下一阶段
- `plan-choice`
  - 用于原计划 / 修订计划 / 信任模式等执行分叉
- `continue-switch-end`
  - 用于继续当前流程、切换模式、结束
- `completion-actions`
  - 用于任务完成后的合并、推送、继续开发等动作

# 增量可测原则

> 所有多步骤任务必须遵守增量可测原则，每步必须产出可独立验证的成果。

## 目录

- 核心要求
- 默认验证顺序
- 判断标准
- 好的拆分示例 (可独立验证)
- 坏的拆分示例 (无法独立验证)
- 重构场景的增量可测
- 优化场景的增量可测
- GUI 场景的增量可测
- 执行原则

## 核心要求

1. 每步完成后，系统必须处于可运行状态
2. 每步必须产出可验证的模块/功能/输出
3. 每步必须有对应的 log 证明其正确性
4. 禁止积攒多步改动到最后验证

## 默认验证顺序

多步骤任务默认按下面的节奏执行：

1. 先验证当前步骤已经完成的后端/规则/API 层
2. 再做对应档位的观测驱动验证
3. 如果当前步骤已经接通 GUI，再补跑本步最小 GUI case
4. 所有步骤完成后，再做一轮功能级整体回归

一句话原则：

- `小步骤增量可测：先测当前层后端，再测本步 GUI。`
- `大功能整体可交付：先回归整体后端，再回归关键 GUI 用户旅程。`

## 判断标准

**问**: 如果只做到这一步就停止，能验证这步是对的吗？

- ✅ 能 -> 步骤拆分合理
- ❌ 不能 -> 步骤拆分有问题

## 好的拆分示例 (可独立验证)

### ✅ Step 1: 实现数据转换函数

- **产出**: transformData() 函数
- **验证**: 单元测试 / log 输入输出
- **可测**: 完全独立

### ✅ Step 2: 实现 API 封装

- **产出**: fetchUserData() 函数
- **验证**: mock 数据测试 / log 请求响应
- **可测**: 完全独立

### ✅ Step 3: 集成到 UI

- **产出**: 完整功能
- **验证**: 先验证 API / 状态，再验证 UI 渲染 + 交互 / log 完整流程
- **可测**: 完全独立

## 坏的拆分示例 (无法独立验证)

### ❌ Step 1: 定义类型和接口

- **问题**: 只有类型定义，无法运行验证
- **正确**: 合并到第一个实现步骤中

### ❌ Step 2: 实现前半部分逻辑

- **问题**: 不完整，无法验证正确性
- **正确**: 拆分成完整的子功能

### ❌ Step 3: 补全剩余逻辑
### ❌ Step 4: 测试整体功能

- **问题**: 积攒到最后验证
- **正确**: 每步都应该可测

## 重构场景的增量可测

### ✅ 正确做法

```text
Step 1: 提取函数 A 并验证等价性
- 产出: 新函数 + 旧函数
- 验证: 对比 log，输出完全一致

Step 2: 替换调用点 X 并验证
- 产出: 更新的调用
- 验证: 功能行为不变，log 一致
```

### ❌ 错误做法

```text
Step 1: 提取 3 个函数
Step 2: 统一测试
- 问题: 无法定位哪个函数有问题
```

## 优化场景的增量可测

### ✅ 正确做法

```text
Step 1: 添加缓存机制
- 产出: 缓存模块
- 验证: log 显示命中/未命中，对比请求次数
```

### ❌ 错误做法

```text
Step 1: 添加缓存
Step 2: 优化算法
Step 3: 测试整体性能
- 问题: 无法分离各优化点的效果
```

## GUI 场景的增量可测

### ✅ 正确做法

```text
Step 1: 打通后端规则和接口
- 验证: 单测 / 集成 / API smoke
- 说明: 只能说“当前层已验证”

Step 2: 接通最小页面交互
- 验证: 后端测试 + 本步最小 GUI case
- 说明: 只验证当前步骤覆盖的用户动作

Step 3: 功能完成
- 验证: 功能级后台回归 + 关键 GUI 用户旅程回归
```

### ❌ 错误做法

```text
Step 1: 改后端
Step 2: 改前端
Step 3: 最后一次性跑完整 GUI
- 问题: 中间每步都不可独立验证，失败时难以归因
```

## 执行原则

- 如果发现步骤无法独立验证 -> **立即停止，重新拆分**
- 如果用户说 "先实现，最后测试" -> **拒绝，解释增量可测的重要性**
- 信任模式下仍需每步可测，只是不等确认

# 测试编排协议

> canonical 测试文档。所有“后台自动测试 / GUI 自治验收 / 手动验收 / 测试回执”都以本文件为准。

## 触发时机（强制）

- 任何代码更新
- 任何测试资产更新
- 任何可能影响运行行为的配置更新

## 核心规则

- **主 Agent 自己验证**，不把基础验证甩给测试 Subagent。
- **代码变更后默认先做后台自动测试**，不再把基础自动测试交给用户选择。
- **任何会改变运行行为的改动，在宣称完成前，必须完成至少一轮对应档位的观测驱动验证**。
- **先过脚本化 Blast Radius 闸门，再组装场景矩阵，再执行测试**。
- **未跑 = 未验证**，不得用“应该没问题”替代测试结果。
- **未覆盖 = 未验证**，未执行的场景必须进入剩余风险，不能笼统说“全部通过”。
- **命中 GUI-capable task 时，默认进入 `GUI 自治验收闭环`**，而不是停在“建议执行”。
- **GUI use case 未达到 `已通过 / 暂不可执行 / 用户禁用 / Manual only` 前，不得宣称完成**。
- **命中 GUI-capable task 时，必须先准备或更新 `.autodev/current-gui-test.js`，并确认它与当前改动直接对应。**
- **历史 GUI/E2E 脚本只能作为补充回归，不能替代当前步骤的 GUI 主验证。**
- **命中大测试时，必须创建或更新 `.autodev/current-test.md`**。
- **业务规则不确定时必须先问**，未确认场景只能标记为“待业务确认”。
- **必须有证据**：命令、输出、截图说明、GUI evidence bundle、或明确的手动检查步骤。
- **命中人工验收 fallback 时，最终结果必须输出 `🧭 开发者手测教程`**。
- **失败重试最多 3 次**，仍失败就停止并报告。

## 四层测试

| 测试层 | 目标 | 默认动作 |
|--------|------|----------|
| 行为场景层 | 把 PM use case 翻译成可执行场景 | 生成场景矩阵，必要时落到 `.feature` 或 `current-test.md` |
| 后台自动测试层 | 验证逻辑、数据流、错误处理、兼容性 | 默认执行单元 / 集成 / 契约 / API smoke |
| GUI 自治验收层 | 验证真实页面 / 窗口流程、跳转、会话、权限、视觉回放 | 命中 GUI-capable task 时默认执行 GUI executor，采集证据并自修复重跑 |
| 人工验收层 | 验证视觉、体感、外部系统、难自动化交互 | 输出具体检查步骤和预期结果；命中 fallback 时生成 `🧭 开发者手测教程` |

## 两种验证节奏

Skill 统一把测试分成两个节奏，避免“每步都像最终验收”或“全部堆到最后再测”。

### 1. Step 级增量验证

适用：Architect / Refactor / Optimize 的逐步执行，或任何可以拆成多个可独立完成步骤的任务。

执行顺序：

1. 第一行代码写入前，先运行本步 Blast Radius，并刷新 `.autodev/current-blast-radius.md`。
2. 先做当前步骤覆盖的后台自动测试。
3. 再做对应档位的观测驱动验证。
4. 若本步已经接通 GUI，则先更新 `.autodev/current-gui-test.js`，再立即补跑本步最小 GUI case。
5. 若 GUI 尚未接通，只能声明“当前层已验证”，不能说整条链路已完成。

推荐表述：

- `先测当前层后端，再测本步已接通的最小 GUI case。`
- `本步未接通 GUI 时，标记为 GUI 暂不可执行，待后续步骤补齐。`

### 2. Feature 级整体回归

适用：关键链路、大测试、跨模块功能、最终交付前的整体验收。

执行顺序：

1. 回归功能级后台自动测试集。
2. 回归关键 GUI 用户旅程。
3. 汇总剩余风险、待业务确认项与人工验收项。
4. 更新 `.autodev/current-test.md` 并输出最终 `🧾 测试回执`。

## 观测驱动验证（强制）

> 具体方法见 `references/principles/observation-driven-verification.md`。本节只定义它与测试编排的关系。

| 档位 | 默认触发 | 最低要求 |
|------|----------|----------|
| `L1` | 一般功能开发、普通 Step、低风险行为改动 | `Happy + 1 个 Negative/Boundary`，明确主观测面与预期观测 |
| `L2` | 多数 Debug、跨模块逻辑、状态流转、权限、缓存、幂等 | 主观测面 + 备用观测面 + 有限 probe + 预期/实际对比 |
| `L3` | Refactor 等价性、疑难 Debug、回归定位、异步链路 | before/after 基线 + 关键路径回归 + 多轮对比 |

关系约束：

- 自动测试负责判断结果。
- 观测驱动验证负责解释过程、校验链路、补抓边界风险。
- 自动测试通过但观测与预期不符，仍不能宣称完成。
- GUI 自治验收负责“像人一样执行 GUI 并完成回放 / 自修复”。

## 场景矩阵（默认补齐）

每个功能或修复至少从以下类型中选择适用项：

- `Happy`：正常路径
- `Negative`：错误输入、拒绝路径、失败提示
- `Zero / Empty`：空值、零值、未填写、无数据
- `Boundary`：阈值、极值、长度边界、页边界
- `Retry / Recovery`：重试、回滚、刷新、恢复
- `Permission / Role`：角色、权限、审批链
- `Manual only`：只能靠人工感知的场景

### AI 可以自动补的 case

- 通用负例
- 空值 / 零值 / 缺省值
- 边界值
- 错误链路
- 兼容旧调用或旧数据格式的常见回归 case

### 必须先问用户的 case

- 失败次数阈值、锁定策略、重试上限
- 角色权限矩阵
- 错误提示是否区分不同原因
- 超时、回滚、审批、幂等、并发业务规则
- 任何“对业务结果有影响，但仓库里没有明确约定”的规则

## 小测试 vs 大测试

| 类型 | 判断标准 | 产物 |
|------|----------|------|
| 小测试 | 单模块、低风险、非关键链路、可用少量自动测试说明白 | `🧾 测试回执` |
| 大测试 | 关键链路、跨模块、契约变更、业务规则不清、需要持续跟踪覆盖 | `.autodev/current-test.md` + `🧾 测试回执` |

### 大测试触发器（满足任一）

- 认证、支付、权限、审批、上传下载
- 多步表单、多页面跳转、会话状态、核心用户链路
- 外部系统联动
- API / 数据契约变更
- 跨模块改动
- 需要向 PM 追问业务规则

## 默认执行顺序

1. 第一行代码写入前运行脚本化 Blast Radius，明确目标文件 / 符号、直接调用方、邻近测试与风险等级。
2. 补数据流 / 直接消费方 / 隐式依赖，写进当前验证范围。
3. 组装场景矩阵，标出 `用户提出 / AI补充 / 待业务确认`。
4. 选定观测驱动验证档位、主观测面、备用观测面。
5. 运行后台自动测试，至少覆盖：
   - 改动点
   - 1 个边界 case
   - 直接影响面中的关键调用方 / 消费方
6. 执行对应档位的观测驱动验证，记录预期 / 实际观测与差异结论。
7. 若当前层已经接通 GUI，先创建或更新 `.autodev/current-gui-test.js`，确认它直接覆盖本次改动，再执行最小 GUI case。
8. 若这是功能级交付或命中大测试，执行一轮功能级回归：
   - 后台回归
   - GUI 关键用户旅程回归
9. 若执行了历史 GUI/E2E 脚本，必须明确标注为“补充回归”，不能把它写成当前改动的主 GUI case。
10. 命中大测试时，更新 `.autodev/current-test.md`。
11. 输出 `🧾 测试回执`；命中人工验收 fallback 时，追加 `🧭 开发者手测教程`。

## 当前 GUI 脚本约定（强制）

- GUI 主测试入口统一使用 `.autodev/current-gui-test.js`。
- 该文件是**当前任务动态准备的测试资产**，不是长期稳定的业务脚本副本。
- 每次准备运行 GUI 前，必须先检查：
  - 当前改动文件/模块是什么
  - 当前改动带来的用户可见风险是什么
  - `.autodev/current-gui-test.js` 的 case 是否直接覆盖这些风险
- 若不能直接回答以上三点，视为“不对应”，必须重写 `.autodev/current-gui-test.js` 后再执行。
- 现有仓库中的 GUI 脚本可以被复用、import 或参考，但最终主入口仍应落到 `.autodev/current-gui-test.js`。
- 只有在 `.autodev/current-gui-test.js` 已通过后，才允许追加更宽的业务 E2E 作为补充回归。

## 统一启动标识协议（强制）

为了让用户和 AI 都能明确“哪一轮 Blast Radius / 后台自动测试 / GUI 自动验收真的开始执行了”，Skill 统一使用同一种启动标识协议。

统一规则：

- 只在**真正开始执行**前输出，不能在计划阶段、建议阶段或尚未执行时提前伪造。
- 启动标识应复用当前任务 / Step / Feature 的业务指纹，便于和存档、日志、case matrix、evidence bundle 对齐。
- 同一轮测试若拆成多条命令，可以共享同一主指纹，必要时追加子序号。
- 同一 GUI case 发生重跑时，应保留同一主指纹，只增加轮次后缀，避免把同一问题误记成多个独立 case。
- 统一格式：

```text
{emoji} {中文动作} - [稳定指纹] {人类可读上下文}
```

- 方括号里的 `稳定指纹` 用来和存档、日志、证据、重跑轮次对齐。
- 花括号里的 `人类可读上下文` 用来让人一眼看懂这轮到底在跑什么；应只放当前最关键的信息，避免塞成长日志。

Blast Radius 启动标识：

```text
💥 Blast-radius 开始 - [BR-{任务指纹}-{Step|Feature}-{targetSlug}-d{depth}] {symbol=... | file=... | mode=... | threshold=... | depth=...}
```

后台自动测试启动标识：

```text
🗄️ 后端测试开始 - [BE-{任务指纹}-{Step|Feature}-{场景ID或场景组}-{测试方式}] {scope=... | layer=... | level=...}
```

GUI 自动验收启动标识：

```text
🖥️ 前端GUI测试开始 - [GUI-{任务指纹}-{Step|Feature}-{caseID}-{executor}-r{轮次}] {scope=主验证/supplemental | visual=headed/headless | gate=GUI}
```

## 后台自动测试（默认执行）

优先使用项目现有测试能力，常见包括：

- 单元测试：纯函数、规则计算、数据转换
- 集成测试：服务层、数据库、依赖协作
- 契约 / API 测试：接口返回、错误码、字段兼容
- 最小调用验证：CLI、`curl`、`node -e`、`python -c`

### 选择原则

- 优先选**最小但足够说明问题**的自动测试集。
- 修 bug 时，优先覆盖“原始复现路径 + 负例 + 边界例”。
- 功能尚未联调完成时，先测试当前已完成层级，不得假装整条链路已验证。
- 新功能开发时，不能只覆盖用户显式提到的 happy path；至少自动补 1 个负例或边界例。
- 真正执行后台自动测试命令前，必须先输出一行 `🗄️ 后端测试开始 - [...] {...}`。

## GUI 自治验收触发器

命中以下任一条件时，AI 必须默认进入 GUI 自治验收闭环：

- 页面流程、核心表单、路由跳转
- 登录 / 注册 / 支付 / 权限 / 会话状态
- 上传下载、搜索筛选、关键用户操作链路
- 任意可点击、可输入、可见结果的 GUI
- 用户显式提到“前端验收”“全链路”“从用户角度”“模拟人类操作”

### GUI executor 选择

- Web GUI → 默认 `Playwright` 且默认 headed
- 其他 GUI → 选择当前环境可用的 GUI driver
- 无自动化入口 → 标记 `Manual only`

### Web GUI 的两种推荐形态

- `Script-first Playwright`
  - 形态：`node agent_test/playwright/foo.ui.test.js`
  - 适用：本地快速闭环、headed 可见执行、AI 按失败结果快速自修复
- `Suite-first Playwright`
  - 形态：`npx playwright test`
  - 适用：CI 并行、统一 report、fixtures、多人协作

两者地位相同；Skill 只要求 `Playwright` 作为 executor，不强制绑定某一种写法。详细 playbook 见 `references/principles/gui-autonomous-loop.md`。

### 证据约束

GUI 自治验收至少保留：

- `action timeline`
- `关键截图`
- `browser console`
- `network trace`
- `页面状态 / DOM 摘要`
- `可关联的后端 trace`（如适用）

### Gate 约束

- `已通过` → 可继续收尾
- `暂不可执行` → 必须说明阻塞原因，并写入剩余风险
- `用户禁用` → 必须在回执和 `current-test.md` 中明确标记
- `Manual only` → 必须输出 `🧭 开发者手测教程`
- `失败修复中 / 规划中 / 执行中` → 不能宣称完成
- 真正拉起 GUI executor 前，必须先输出一行 `🖥️ 前端GUI测试开始 - [...]`。

## 开发者手测教程触发器

命中以下任一条件时，最终结果必须追加 `🧭 开发者手测教程`：

- 场景矩阵中存在 `Manual only`
- `.feature` 或计划中存在 `@manual`
- GUI 自治验收状态为 `用户禁用`、`暂不可执行` 或 `Manual only`
- 关键观测面需要开发者本地环境、私有 IDE / Figma 控制台、真机、外设、第三方私有后台

要求：

- 教程必须是开发者可直接执行的 step-by-step 指南，不能只写“手动验证一下”
- 必须写清前置条件、操作步骤、每步预期结果、最小回传证据
- 若当前只能验证部分链路，教程必须明确“本轮已验证到哪一层、剩余哪一层待人工确认”

- 结果: [通过 / 失败 / 无法执行]
- 证据: [关键输出]

- 档位: [L1 / L2 / L3]
- 主观测面: [...]
- 备用观测面: [...]
- 预期观测: [...]
- 实际观测: [...]
- 差异结论: [通过 / 失败 / 暂不可执行 / 待业务确认]
GUI 自治验收:
- 状态: [未触发 / 规划中 / 执行中 / 已通过 / 失败修复中 / 暂不可执行 / 用户禁用 / Manual only]
- 启动标识: [🖥️ 前端GUI测试开始 - ... / 无]
- Executor: [...]
- 可视化执行: [required / preferred / unavailable]
- 覆盖用例: [...]
- 证据: [timeline / screenshot / console / network / trace]

- 预期结果: [无 / 列出]

```text

操作步骤:
1. [Step 1]
2. [Step 2]
3. [Step 3]
每步预期:
- Step 1 → [...]
- Step 2 → [...]
- Step 3 → [...]

```

### 大测试附加要求

- `.autodev/current-test.md` 中必须同步更新：
  - 场景矩阵状态
  - 执行记录
  - 待业务确认问题
  - 剩余风险

## 失败处理

1. 先修代码，再重试验证。
2. 再检查测试方法是否本身有误。
3. 连续 3 次失败，停止推进并请求人工介入。
4. 若业务规则不明确，停止“继续补测”，改为向用户提问。

## 禁止行为

- 跳过后台自动测试直接说“已验证”。
- 只有结论，没有命令 / 输出 / 手动步骤。
- 只覆盖 happy path 就说“功能测试全部通过”。
- 命中大测试却不创建 / 更新 `current-test.md`。
- 命中人工验收 fallback，却只写“请手动验证”而不给可执行教程。
- 命中 GUI-capable task 却只写“建议执行”而不真正执行。
- 把 `Playwright passed` / `GUI passed` 说成“真实用户上线一定没问题”。
- 把用户暂未确认的业务规则当成既定事实写进测试结论。
- 伪造测试输出或把验证责任甩给用户。

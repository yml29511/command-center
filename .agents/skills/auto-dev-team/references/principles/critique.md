# 自动会诊机制

> ⚕️ 默认 strict：每个写入方案都需要第二意见。能用 Subagent 时优先独立会诊，不能用时退化为本地 checklist，但会诊能力不能消失。

## 目标

- 对抗 confirmation bias
- 发现需求歧义、过度设计、隐藏副作用
- 保证“先澄清，再执行”
- 保留小任务节奏：如果环境不支持 Subagent，不阻塞流程

## 默认触发时机

`.autodev/autodev-config.json` 中 `critique.threshold=all_write_modes` 时，以下场景默认触发：

- 新功能开发方案
- Debug 修复方案
- 重构 / 优化方案
- Hotfix / FastTrack 这类小改动方案

若项目把阈值调高，仍必须至少保留“本地 checklist 审查”。

## 执行流程

```text
1. 主 Agent 生成方案
2. 优先调用 Critique Subagent
   - 传递：【用户原始需求】+ 方案内容 / 诊断结论 / 修复方案 / 依据
   - 不传递：主 Agent 的思考过程
3. 若环境不支持 Subagent：
   - 主 Agent 按下方 checklist 做一次显式本地会诊
   - 输出原计划 / 修订计划 / 是否需要用户澄清
4. 会诊分两阶段：
   - 阶段 A：需求澄清检查
   - 阶段 B：方案审查
5. 若阶段 A 发现问题，立即暂停，先问用户
6. 若阶段 B 通过，输出原计划与修订计划（如有）
7. 默认要求用户选择后再执行
```

短模板见 `references/shared/flow-snippets.md`。

## 阶段 A：需求澄清检查

| 检查项 | 检查内容 |
|--------|----------|
| 需求合理性 | 用户要做的事情合理吗？有没有更直接的方案？ |
| 需求歧义 | 是否有多种实现方式会导致后续分叉？ |
| 信息完整性 | 是否缺少边界、约束、优先级、验收标准？ |
| 风险预警 | 用户是否知道副作用、成本或影响面？ |

⛔ 阶段 A 有问题时：必须暂停并向用户澄清，不能直接执行。

## 阶段 B：方案审查

| 检查项 | 检查内容 |
|--------|----------|
| 🔬 根因验证 | 确认了还是在猜？基础检查做了吗？ |
| 🎯 过度设计 | 有没有 PM 没要求的东西？ |
| 💰 成本顺序 | 是否优先尝试免费且简单的方案？ |
| ⚡ 副作用 | 会引入新问题吗？可回退吗？ |
| 🛡️ 保留性 | 用户说“添加”时是否做成了“替换”？是否误删同级/内层内容？ |
| 🔗 关联性 | 是否有对称操作被遗漏？完整链路是否都检查了？ |
| 📐 继承性 | 改了基类/接口，子类/实现是否需要同步？ |
| 🧱 代码结构 | 是否在单文件堆积多种职责？新功能是否该融入现有模块？ |

## Subagent 与降级策略

- 优先使用项目级 `.cursor/agents/critique.md`
- 其次使用用户级 `~/.cursor/agents/critique.md`
- 若当前环境不能启 Subagent：
  - 主 Agent 显式执行一次本地 checklist
  - 仍要输出“原计划 / 修订计划 / 待澄清问题”
  - 默认仍等待用户选择

## 模式集成

各写入模式在方案输出后，统一按本节执行会诊。模式 README 只需写：

```text
⭐ 方案输出后，按 `references/principles/critique.md` 的模式集成流程执行自动会诊。
本模式审查重点：[见下表]
```

### 各模式审查重点

| 模式 | 传递内容 | 阶段 A 澄清重点 | 阶段 B 审查重点 |
|------|----------|------------------|-----------------|
| **Architect** | 用户原始需求 + 开发方案 | 需求是否合理？有更好方式？缺关键信息？ | 过度设计？成本？副作用？ |
| **Refactor** | 用户重构目标 + 重构方案 | 重构目标是否明确？范围是否需确认？隐含业务约束？ | 是最简方案吗？会不会改太多？更安全的方式？ |
| **Optimize** | 用户优化诉求 + 优化方案 | “慢”是什么场景？有量化数据？优化目标明确？ | 过早优化？更简单方案？会影响功能？ |
| **Debug** | 用户问题描述 + 确诊结论 + 治疗方案 | 症状描述是否完整？需更多复现信息？ | 根因是否确认？是否对症？副作用？ |
| **Hotfix** | 用户问题描述 + 最小修复方案 | 问题描述是否足够定位？（快速版，最多 1-2 个问题） | 是否真的是最小改动？会引入新问题？ |
| **FastTrack** | 用户原始请求 + 快速方案 | 请求是否明确？（快速版，最多 1 个问题） | 是否真的是小改动？方案是否正确？ |

## 通用禁止规则

- 主 Agent 自己脑补“已经审过了”就跳过会诊
- ⛔ 禁止跳过会诊直接让用户选择执行
- ⛔ 禁止会诊后直接执行，必须等待用户选择
- ⛔ 禁止在需求有澄清问题时输出方案选项

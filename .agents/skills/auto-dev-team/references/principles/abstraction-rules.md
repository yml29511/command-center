## 三问法则 (抽象前必问)

### Q1: 这个功能是"中性动作"还是"业务特化"？

| 类型 | 特征 | 抽象价值 |
|------|------|----------|
| **中性动作** | 扫描、遍历、转换、校验、格式化... | ✅ 有抽象价值 |
| **业务特化** | 用户画像计算、订单状态机、特定UI组件... | ❌ 通常不需要 |

### Q2: 未来会有其他场景使用吗？

| 判断 | 结论 |
|------|------|
| 🟢 明确会有 (如"扫描"可用于字符/形状/图片) | → **抽象** |
| 🟡 可能会有但不确定 | → **先不抽象，标记"潜在抽象点"** |
| 🔴 不会有 | → **不抽象** |

### Q3: 抽象后 API 会更简单还是更复杂？

| 判断 | 结论 |
|------|------|
| 更简单 (如 `scan(target, type)`) | → **抽象** |
| 更复杂 (参数爆炸、泛型地狱) | → **不抽象，这是过度工程** |

## Rule of Three (事后抽象)

| 出现次数 | 策略 |
|----------|------|
| 1-2 次 | 不抽象，允许重复 |
| 3+ 次 | **必须抽象** |

## 抽象决策流程图

```
发现可能的抽象机会
        │
        ▼
    Q1: 中性动作？
        │
   ┌────┴────┐
   │         │
  是        否 → 不抽象
   │
   ▼
Q2: 其他场景会用？
   │
   ├─ 明确会 → Q3
   ├─ 不确定 → 标记观察
   └─ 不会 → 不抽象
   │
   ▼
Q3: API 更简单？
   │
   ├─ 是 → ✅ 抽象
   └─ 否 → ❌ 过度工程
```

## 抽象产出

抽象完成后必须:
1. 新 Utility 记录到 `module-registry.md`
2. 更新 `project-map.md` 标注新的通用能力

输出: `📝 已自动更新: module-registry.md - 新增 [Utility名]`

## 常见抽象陷阱

### ❌ 过早抽象
```
只出现 1 次就抽象
→ 浪费时间，可能抽象方向错误
```

### ❌ 过度泛化
```
为了"通用"加入大量配置参数
→ API 复杂度爆炸，不如不抽象
```

### ❌ 业务逻辑抽象
```
把订单计价逻辑抽象成"通用计价器"
→ 业务规则千变万化，抽象反而是负担
```

## 好的抽象示例

```typescript
// ✅ 中性动作，多场景复用，API 简单
function traverse<T>(
  root: T, 
  getChildren: (node: T) => T[]
): T[] { ... }

// 使用场景:
// - 遍历文件树
// - 遍历 DOM 节点
// - 遍历组织架构
```

## 不该抽象的示例

```typescript
// ❌ 业务特化，只有一个场景
function calculateUserVIPDiscount(user, order) {
  // VIP 折扣规则经常变，抽象没意义
}
```

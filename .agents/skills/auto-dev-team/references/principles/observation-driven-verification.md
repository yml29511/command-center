# 观测驱动验证

> canonical 观测验证文档。定义“预期观测 vs 实际观测”的验证方法，适用于新功能、Step、Debug、Refactor、Tester 与回归定位。

## 适用范围

观测驱动验证不是 Debug 专属能力。凡是会改变运行行为的改动，都应按风险启用对应档位：

- 新功能开发
- Step 执行中的行为改动
- FastTrack / Hotfix 的行为修复
- Debug 根因诊断
- Refactor 等价性验证
- Tester 在补测试或定位根因时
- 回归定位：不确定哪次提交引入 bug

## 核心原则

- **先定义预期观测，再运行验证**。
- **自动测试判断结果，观测验证解释过程**。
- **未覆盖 = 未验证**，未执行的场景必须进入剩余风险。
- **AI 自己采证优先**，不能默认把日志采集工作甩给用户。
- **GUI-capable task 由 `references/principles/gui-autonomous-loop.md` 负责执行闭环；观测驱动验证负责定义观测点并解释差异。**
- **不追求全量重模式**，按风险选择 `L1 / L2 / L3`。

## 三档策略

| 档位 | 适用场景 | 最低要求 |
|------|----------|----------|
| `L1 轻量观测验证` | 一般功能开发、普通 Step、低风险行为改动 | 明确场景、预期结果、1-2 个关键观测点，完成 1 轮最小验证 |
| `L2 标准观测验证` | 多数 Debug、跨模块逻辑、状态流转、权限、缓存、幂等 | 主观测面 + 备用观测面 + 有限 probe + 预期/实际对比 |
| `L3 重度观测验证` | Refactor 等价性、疑难 Debug、回归定位、异步链路 | before/after 基线 + 多轮观测 + 关键路径回归 + 差异归因 |

## 模式默认档位

- Architect：计划阶段标注档位；执行时按 Step 生效
- Step / FastTrack / Hotfix / Tester：只要行为改动，默认至少 `L1`
- Debug：默认 `L2`
- Refactor：默认 `L3`
- 回归定位：默认 `L3`

## 观测面

Skill 统一使用“观测面”而不是“console log”。

| 类型 | 用途 |
|------|------|
| `browser_console` | 浏览器前端、Figma/插件前端、页面脚本 |
| `process_stdout` | Node / Python / Go / CLI / 本地服务输出 |
| `app_log_file` | 服务端文件日志、worker 日志、容器日志 |
| `test_runner_output` | `pytest` / `vitest` / `jest` / `go test` 等测试输出 |
| `network_trace` | 请求是否发出、参数、状态码、响应体 |
| `artifact_snapshot` | 数据库状态、DOM 快照、生成文件、JSON 输出 |

选择原则：

1. 优先选择 AI 可直接访问、成本最低、最贴近故障边界的观测面。
2. 至少标注一个备用观测面，主观测面不可达时可降级。
3. 涉及运行时路径、日志路径、控制台入口时，先读 `.autodev/path.md`。
4. 若当前任务为 GUI-capable task，优先让观测面贴近 GUI executor 的证据包（timeline / screenshot / console / network）。

## 最低覆盖要求

AI 不能只验证用户明确提到的 happy path。至少要自动补齐以下场景：

- `Happy`
- `Negative` 或 `Boundary` 至少 1 个
- 修 bug 时：原始复现路径 + 1 个边界或负例
- Refactor 时：原始调用路径 + 关键副作用路径

以下场景可由 AI 自动补：

- 通用负例
- 空值 / 零值 / 缺省值
- 边界值
- 重试 / 恢复
- 兼容旧调用 / 旧数据格式的常见回归 case

以下场景必须先问用户：

- 失败阈值、锁定策略、重试上限
- 权限矩阵
- 影响业务结果的阈值与审批规则
- 仓库内没有明确约定的业务规则

## 标准流程

1. 先组装场景矩阵，标记 `用户提出 / AI补充 / 待业务确认`。
2. 选定当前档位、主观测面、备用观测面。
3. 写出预期观测：
   - 会进入哪些关键节点
   - 哪些关键变量或状态会变化
   - 哪条分支应被命中
   - 会产生哪些输出或副作用
4. 必要时插入临时 probe，统一 fingerprint。
5. AI 自己运行最小复现、自动测试、CLI，或在 GUI-capable task 中执行 GUI executor。
6. 采集实际观测并与预期对比。
7. 输出结论：
   - 通过
   - 失败
   - 暂不可执行
   - 待业务确认
8. 若失败：修复后再跑同一组场景。
9. 完成后清理临时 probe，仅保留必要基础观测。

## 与 GUI 自治验收闭环的关系

- 观测驱动验证不是 GUI executor 的替代品。
- GUI 闭环负责“像人一样执行界面并回放过程”。
- 观测驱动验证负责“为 GUI case 写出预期观测、解释实际差异、帮助归因到视觉 / 状态 / 网络 / 后端”。
- GUI 自动执行通过但观测与预期不符，仍不能宣称完成。

## 预期观测写法

避免写成大量散乱日志。优先写“结构化观测点”：

```text
场景: S2 重复提交
主观测面: process_stdout
预期观测:
- 命中 duplicate_check
- create_order 不应执行
- 返回 code=409
- 不应产生第二次写库副作用
```

若需要 probe，使用统一指纹，例如：

- `[DEV-{主题}-Step{N}]`
- `[DEBUG-{主题}-R{N}]`
- `[REFACTOR-{主题}-before]`
- `[REFACTOR-{主题}-after]`

probe 只记录定位所必需的信息：

- 进入节点
- 关键变量
- 分支命中
- 输出 / 副作用

## GUI 观测模板

对于 GUI case，推荐把预期观测固定写成三层，而不是只写“页面应该成功”：

```text
场景: G3 上传成功
主观测面: network_trace
备用观测面: browser_console / artifact_snapshot
预期观测:
- 页面变化:
  - 上传弹窗关闭
  - 列表出现新条目
- 网络行为:
  - POST /api/upload 返回 200
  - 响应体 success=true
- 后端副作用:
  - 生成文件记录
  - 存储中出现对应对象
- 失败时优先观测面:
  - network_trace -> browser_console -> backend_trace
```

GUI 预期观测至少优先覆盖：

- `页面变化`
- `网络行为`
- `后端副作用`
- `失败时优先观测面`

## GUI 通过但观测不通过

以下情况仍视为未完成：

- 页面看起来通过，但关键请求参数错误
- 页面看起来通过，但服务端副作用没有发生
- Playwright 通过，但 headers / trace / DOM 状态与预期不符

换句话说：

- GUI 自动执行通过 != 观测通过
- 观测通过 != 业务一定正确
- 两者都满足，才可以宣称闭环完成

## 差异分类

对比预期与实际时，至少归到以下一类：

- **缺失**：预期的观测点没有出现
- **错序**：顺序不对
- **值不符**：变量、状态、返回值与预期不同
- **分支错位**：命中了错误分支
- **多余副作用**：多写库、多发请求、多创建资源
- **暂不可判定**：观测面不足或业务规则未确认

## 清理与保留

- 临时 probe 默认在验证完成后删除。
- 若某个观测点对长期可观测性有价值，可精简后保留为基础日志。
- 禁止把所有临时 `[DEBUG-*]` / `[DEV-*]` / `[REFACTOR-*]` 探针永久留在代码中。

## fallback 机制

AI 必须先尝试自己采证，只有在观测面不可达时才能请求用户协助。

允许请求用户补充证据的典型情况：

- 私有 IDE / Figma 控制台
- AI 无权限访问的远端日志
- 真机 / 外设 / 硬件
- 第三方平台私有后台

请求方式必须最小化：

- 说明要哪一个观测面
- 说明要哪一段最小必要证据
- 用户补一次后，AI 继续自己分析，不能反复让用户搬运日志

## 不适用场景

以下问题不能只依赖观测驱动验证：

- 纯视觉问题
- 纯交互体感问题
- 性能抖动、内存泄漏、profiling
- 安全漏洞
- 生产环境专属差异

这些问题需要结合截图、GUI 自治验收闭环、人工验收、benchmark、trace、专项安全测试等手段。

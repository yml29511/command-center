# BDD / 行为场景规范

> BDD 在本 Skill 中首先是“行为场景合同”，其次才是 `.feature` 文件。目标是让 PM 看得懂、AI 跑得稳、测试资产能落地。

## 目录

- 核心定义
- 行为场景包最小结构
- 场景来源
- AI 补场景规则
- 场景落层映射
- `.feature` 文件约定（项目有 BDD 框架时）
- 何时用 `.feature`
- 常见框架与命令

## 核心定义

- `行为场景`：描述用户目标、前置条件、触发动作、预期结果。
- `BDD 资产`：可以是 `.feature`、测试用例说明、`current-test.md` 里的场景矩阵。
- `.feature`：当项目已配置 Cucumber / behave / godog 等框架时的优先落地形式。

## 行为场景包最小结构

每个场景至少写清楚：

- 用户目标
- 前置状态
- 触发动作
- 预期结果
- 异常 / 错误表现
- 边界条件
- 是否需要人工验收

## 场景来源

| 来源 | 处理方式 |
|------|----------|
| 用户提出 | 必须保留 |
| AI 补充的通用负例 / 边界例 | 默认加入 |
| 待业务确认的规则型场景 | 先提问，未确认前标记为 `待业务确认` |

## AI 补场景规则

允许 AI 主动补：

- 错误输入
- 空值 / 零值 / 缺省值
- 边界值
- 失败提示
- 重试 / 恢复
- 兼容旧数据 / 旧调用的回归场景

不允许 AI 擅自决定：

- 锁定阈值
- 权限矩阵
- 超时 / 重试 / 审批策略
- 对业务结果有直接影响的阈值和规则

## 场景落层映射

| 场景类型 | 优先落层 |
|----------|----------|
| 纯规则、纯函数、数据转换 | 单元测试 |
| 服务协作、数据库、接口响应 | 集成 / 契约测试 |
| 页面流程、跳转、会话、表单链路 | GUI 自治验收 |
| 视觉、动效、外部系统、复杂人工感知 | 人工验收 |

### 登录功能示例

| 场景 | 推荐测试层 |
|------|-----------|
| 登录成功 | 单测 / 集成 + GUI 自治验收 |
| 密码错误 | 单测 / 集成 + GUI 自治验收 |
| 空密码提交 | 单测 / GUI 页面交互验证 |
| 连续输错后是否锁定 | 先问业务，再决定测试实现 |
| 错误文案视觉样式 | 人工验收或 GUI 截图 |

## `.feature` 文件约定（项目有 BDD 框架时）

```text
{项目根}/features/
  ├── {功能名}.feature
  └── steps/
      └── {功能名}.steps.*
```

### `.feature` 文件规则

- 一个 Feature = 一个功能主题
- 一个 Scenario = 一个独立行为
- Given = 前置状态 / When = 触发动作 / Then = 预期结果
- 通用负例和边界例可以由 AI 主动补充
- 无法自动化的场景标注 `@manual`
- 若业务规则未确认，在场景标题或注释中标注 `待业务确认`

### Step Definitions 规则

- 一个 step 写一次，跨场景复用
- Mock 外部依赖时只 mock 行为，不 mock 实现细节
- 断言失败 = 场景失败
- Step definitions 只是实现层，不能替代场景合同本身

## 何时用 `.feature`

| 场景 | 建议 |
|------|------|
| 项目已配置 BDD 框架，且场景有明确前置→触发→结果 | 用 `.feature` |
| 项目没有 BDD 框架，但仍需沉淀行为场景 | 写进 `current-test.md` 或计划文档 |
| 只是一两个快速规则验证 | 直接落单测 / 集成测即可 |

## 常见框架与命令

| 语言 | BDD 框架 | 命令 |
|------|---------|------|
| JS/TS | `@cucumber/cucumber` | `npx cucumber-js` |
| Python | `behave` | `behave` |
| Go | `godog` | `godog` |
| Java | `cucumber-jvm` | `mvn test` |
